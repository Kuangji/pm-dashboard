## 1. 模块综述

Nox 当前站点应被理解为一个由 **公开获客入口、发现与情报工作台、运营执行工作台、平台治理与服务底座** 组成的复合平台，而不是一个单一“达人搜索工具”。

它的主要价值不只在“找到谁”，还在于：

- 持续沉淀合作对象
- 组织批量触达与实时沟通
- 监控发布与效果
- 处理支付结算
- 对额度、权限、安全与新能力做平台级治理

- 找达人

### 2.3 运营执行层

- 沉淀对象

主要模块：

- 数据用量与配额

## 3. 核心名词定义

| 名词 | 定义 |
|------|------|
| **公开获客层** | 无需登录即可访问的产品叙事、内容教育和转化入口 |
| **发现与情报层** | 以达人、商品、品牌、竞品为对象的情报发现层 |
| **运营执行层** | 把已发现对象转成可执行合作流程的工作台层 |
| **平台治理与服务层** | 为多个业务模块提供额度、权限、安全、通知、API Key 等治理能力 |
| **达人搜索工作台** | 以频道、视频、筛选条件为核心的发现入口 |
| **品牌情报** | 以品牌和竞品为核心的分析与洞察能力 |
| **收藏夹 / 资源夹** | 轻量沉淀和分拣区，不等于正式 CRM |
| **网红 CRM** | 网红关系主档与合作状态管理区 |
| **邮件管理** | 项目化外发触达与邮件统计区 |
| **消息中心** | 已建立联系后的实时沟通区 |
| **内容监控 / 自动追踪** | 对已发布内容或规则对象进行持续监控的执行模块 |
| **数据用量** | 多个高成本能力的使用记录与剩余额度台账 |
| **Skills** | Nox 站内的新能力控制台，包含 API Key、用量与账单等面板 |

## 4. 典型业务主链路

后续做模块评价或竞品对照时，默认采用这条主链路：

> **找达人/商品/品牌情报 -> 建档与沉淀 -> 批量触达 -> 实时沟通 -> 监控发布效果 -> 支付结算 -> 配额/权限/平台治理**

对应模块大致为：

| 链路阶段 | 对应模块 |
|------|------|
| 发现 | 搜索工作台、TikTok Shop 搜索、品牌分析、品牌榜单、产品评论洞察 |
| 沉淀 | 收藏夹、CRM |
| 触达 | 邮件管理 |
| 沟通 | 消息中心 |
| 执行编排 | 智能营销计划 |
| 监控 | 内容监控、自动追踪 |
| 结算 | 支付中心 |
| 治理 | 数据用量、通知、权限、安全、Skills |

## 5. 模块边界与不要混淆的概念

### 5.1 搜索 vs 品牌分析

- 搜索：从达人/商品/内容对象出发
- 品牌分析：从品牌/竞品对象出发

### 5.2 收藏夹 vs CRM

- 收藏夹：轻量沉淀
- CRM：正式关系主档

### 5.3 邮件管理 vs 消息中心

- 邮件管理：项目化批量触达
- 消息中心：实时对话处理

### 5.4 品牌分析 vs 内容监控

- 品牌分析：策略与市场洞察
- 内容监控：项目级持续追踪

### 5.5 TikTok Shop 搜索 vs 商品中心 vs 产品评论洞察

- TikTok Shop 搜索：发现外部机会
- 商品中心：内部商品协同
- 产品评论洞察：消费者评论理解

### 5.6 智能营销计划 vs AI 搜索 vs AI Assistant

- 智能营销计划：登录后可见的业务计划模块
- AI 搜索：公开层 AI 入口
- AI Assistant：本轮实勘中为 blocked page，不能直接当作稳定能力

## 8. 使用规则

### 8.1 什么时候先读这份文档

以下任务开始前，默认先读本文件：

- 站级调研
- 模块评价
- 竞品对照
- 新需求定位
- 跨模块体验梳理

### 8.2 如何和模块级文档配合

- 先用本文件判断模块所在层级与主链路位置
- 再进入更细的模块级 `common_context`
- 如果模块级文档与本文件冲突，以更细、更近、更新的模块级文档为准

### 8.3 更新规则

只有在以下情况才建议更新本文件：

1. 出现新的一级模块
2. 主链路出现明显变化
3. 受限路由变成稳定能力
4. 公开层与工作台层的边界发生重组
