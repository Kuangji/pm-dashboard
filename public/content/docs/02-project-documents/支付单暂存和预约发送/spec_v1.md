## 一、功能概述

本次迭代包含两个独立功能，同期上线：

1. **支出单暂存（草稿）**：允许广告主在创建支出单过程中将未填完的表单保存为草稿，后续可继续编辑或直接提交。
2. **预约发送**：允许广告主在提交支出单时选择「预约发送」，系统在满足前置条件后自动在指定时间触发 B-b→B-c 转账流程。

**与现有状态机的关系**：

- 草稿存储于表单层，独立于状态机之外，不占用任何状态码。草稿提交后才正式进入状态机（状态 A）。
- 预约发送不引入新的**对外状态码**，但在 B→B1 之间引入内部逻辑状态 **B-b（预约等待中）**（详见 §四）。

> ⚠️ **易混淆点**：现有状态机中「状态 A」名称为「暂存支付单，等待合同签字完毕」，这是**正式提交后**的状态。本功能的「草稿」独立于状态机之外，两者不同，文档中明确区分。

## 二、功能一：支出单暂存（草稿）

### 2.1 触发条件与保存逻辑

#### 最低可保存条件

**网红频道字段已选择**（其他字段可以为空）。

#### 触发方式 A — 主动保存

- 创建表单内显示「保存草稿」按钮
- 满足最低条件时按钮可点击
- 未满足时按钮置灰，鼠标悬停显示 tooltip：「请先选择网红频道」

#### 触发方式 B — 关闭弹窗时系统提示

用户点击弹窗关闭（×）或「取消」按钮时，根据场景给出不同提示：

**场景一：全新创建时关闭（二选）**

| 场景 | 行为 |
|------|------|
| 已选择网红频道（满足最低条件） | 弹出确认弹窗：「是否保存草稿？」，操作：「保存草稿」/「不保存，直接关闭」 |
| 未选择网红频道（不满足最低条件） | 直接关闭弹窗，无任何提示 |

**场景二：「继续编辑」已有草稿后关闭（三选）**

| 条件 | 行为 |
|------|------|
| 有修改未保存 | 弹出确认弹窗：「尚未保存修改，是否保存？」，操作：「保存草稿」/「不保存，恢复到上次保存」/「取消」 |
| 无修改 | 直接关闭弹窗，无提示 |

> 「取消」按钮语义：用于误操作兜底，点击后留在弹窗内继续编辑，不执行关闭。与「不保存，恢复到上次保存」不同——后者关闭弹窗并回滚，前者保持弹窗开启。

### 2.2 草稿存储规则

| 规则项 | 值 |
|--------|-----|
| 存储层 | 后端持久化（可跨设备恢复，非 localStorage） |
| 有效期 | 永不自动过期，用户手动删除 |
| 单账号数量上限 | 20 条（超出时阻止新建，提示清理） |
| 可见范围 | 仅创建人（同账号其他成员不可见） |
| 保存内容 | 仅保存表单填写字段；**不含发送时间**（发送时间属于提交操作，见 §3.1） |

### 2.3 草稿在列表中的展示

#### 状态标签

列表新增「草稿」状态标签，与现有「进行中」「已取消」并列。标签颜色/图标样式由 Design 阶段定义。

#### 列表位置与排序

- 草稿条目**置顶展示**，排在所有正式支付单之前。
- 多条草稿之间，**按最后编辑时间倒序排列**（最近编辑的草稿排在最前）。

#### 筛选支持

现有「支出状态」筛选下拉新增「草稿」选项，支持单独筛选草稿条目。

#### 列表行展示字段

| 字段 | 说明 |
|------|------|
| 网红头像 + 频道名 | 必填，必有 |
| 其他已填字段（币种、金额等） | 未填写的字段显示「—」 |
| 状态标签 | 「草稿」 |
| 进度时间线 | **不显示**（时间线从正式提交后才出现） |
| 操作按钮 | 「继续编辑」/「删除」 |

#### 操作业务规则

| 操作 | 触发条件 | 行为 | 不可用场景 |
|------|---------|------|-----------|
| 继续编辑 | 随时可用 | 打开创建弹窗，恢复所有已保存字段内容；用户可再次保存草稿或直接提交 | 无 |
| 删除 | 随时可用 | 弹出二次确认「删除后无法恢复，确认删除？」；确认后草稿记录永久删除 | 无 |
| （隐式）草稿自动清除 | 草稿提交成功后 | 草稿记录自动删除，不在列表中保留，无需用户操作 | 仅在提交成功时触发 |

### 2.4 草稿与正式支付单的转换关系

```
草稿（表单层）
  → 用户点击「立即发送」或「预约发送」
  → 正式创建支付单
  → 进入状态 A（等待合同签字）
  → 草稿记录自动删除
```

草稿阶段与状态机完全隔离，提交前对状态机零影响。

## 三、功能二：预约发送

### 3.1 提交区域的发送时间选项

> ⚠️ **架构说明**：「发送时间」**不是表单填写字段**，而是**提交操作的一部分**。「保存草稿」与「立即发送」/「预约发送」互斥，草稿中不保存发送时间。

#### 提交区域结构

创建支出单表单底部，收款信息和金额填写完成后，操作区并列展示三个操作：

| 操作 | 性质 | 说明 |
|------|------|------|
| 保存草稿 | 次要操作 | 仅保存表单内容，不触发支付流程 |
| 立即发送 | 主要操作（默认） | 提交支付单，审核通过后即时触发转账 |
| 预约发送 | 主要操作变体 | 提交支付单，审核通过后等待预约时间到达再触发转账 |

#### 「预约发送」交互

用户选择「预约发送」后，展开日期 + 小时选择器，确认时间后正式提交。

| 约束项 | 规则 |
|--------|------|
| 最早可选时间 | 创建时间 + 72 小时（3 天后）；不足 3 天的时间段置灰不可选 |
| 最晚可选时间 | 无硬性限制（UI 建议默认展示未来 3 个月内） |
| 时间精度 | 精确到小时（如：2026-03-10 14:00） |
| 时区 | 平台服务器时区（UI 上需标注当前时区） |

#### 表单内固定提示文案

> 请在预约时间的 24 小时前完成网红签合同，否则审核可能无法完成，导致预约失败

（固定展示于时间选择器下方，非弹窗）

### 3.2 预约状态在列表中的展示

支付单处于 **B-b 阶段（预约等待中，见 §四）** 时：

- 进度时间线第三节点显示：「开始转账（预约中）」+ 预约时间
- 操作按钮区域显示：「修改预约时间」/「转为立即发送」

### 3.3 成功路径：自动触发逻辑

系统定时任务轮询，同时满足以下两个条件时自动触发 B-b→B-c：

```
AND 条件：
  ① 支付单已到达 B-b（预约等待中）
  ② 当前时间 ≥ 预约发送时间
→ 系统自动执行 B-b→B-c（开始转账）
→ 进度时间线恢复正常流转
```

### 3.4 失败路径：预约失败处理

#### 触发条件

预约时间到达时，支付单**尚未到达 B-b**（合同未签或审核未通过）。

#### 失败原因分类

| 原因类型 | 触发条件 | 通知文案要点 |
|---------|---------|------------|
| 原因 A：合同未完成签署 | 预约时间到达时，支付单仍在状态 A（合同签署未完成） | 说明合同未签，引导用户催促网红尽快完成签署 |
| 原因 B：审核未在截止前通过 | 预约时间到达时，支付单仍在状态 A1（合同已签但审核未通过） | 说明平台审核问题，引导用户联系客服或重新安排 |

#### 系统动作

1. 标记该支付单为「预约失败」
2. 立即向广告主发送站内信通知（内容：**具体失败原因**（按上述原因 A/B 分类）+ 三种可选操作引导）

#### 用户可选的三种处理方式

| 操作 | 说明 | 执行后状态 |
|------|------|-----------|
| 重新设置预约时间 | 复用原支付单 ID，仅修改预约时间；**最早可选时间 = 当前操作时刻 + 72 小时** | 重新进入预约等待（B-b） |
| 转为立即发送 | 取消预约，一旦前置条件（到达 B-b）满足，立即自动触发 B-c | 恢复正常流转 |
| 作废 | 支付单进入取消态（等同于 A2/A3），不可恢复 | 终止 |

### 3.5 预约修改与取消规则

| 操作 | 状态 A | 状态 A1 | 状态 B-b | 状态 B-c 及之后 |
|------|:------:|:-------:|:--------:|:--------------:|
| 修改预约时间 | ✅ | ❌ | ✅（列表按钮） | ❌ |
| 取消预约（转为立即发送） | ✅ | ✅ | ✅（列表按钮） | ❌ |

**规则说明**：
- 状态 A1 后**不可**修改预约时间：审核启动后改时间会导致审核结果与预约时间不对应，逻辑混乱。
- 状态 A/A1 的「转为立即发送」操作入口（不同于 B-b 的列表按钮），具体 UI 交互方案待 Design 阶段定义。
- 预约失败后的「重新设置预约时间」参见 §3.4，最早可选基准为**当前时间 + 72h**（非原创建时间）。

### 3.6 提醒机制

| 提醒类型 | 触发时机 | 渠道 | 内容要点 |
|---------|---------|------|---------|
| 预约前提醒 | 预约时间前 48 小时 | 站内信 | 提醒预约即将到达，请确认合同已完成签署 |
| 预约失败通知 | 预约失败即时触发 | 站内信 | 具体失败原因（按原因 A/B 分类文案）+ 三种可选操作入口 |

> **说明**：草稿无到期机制（永不过期），因此无草稿过期提醒。

## 四、对现有状态机的影响

### 状态 B 的拆分说明

原状态 B（「审核通过，等待支付通道处理」）在引入预约发送后，需区分两种情形：

| 内部阶段 | 含义 | 预约支付单 | 立即发送支付单 |
|---------|------|-----------|--------------|
| B-a | 审核通过，系统判断是否有预约 | 进入 B-b | 直接进入 B-c |
| B-b | **预约等待中**（等待预约时间到达） | 存在此阶段 | 无此阶段 |
| B-c | 等待支付通道处理（即原 B1） | 由 B-b 触发 | 由 B-a 直接触发 |

> B-a/B-b 是内部逻辑状态，**不一定作为新对外状态码**暴露；是否新增状态码由后端开发决定。
> Product 侧要求：**列表层面对 B-b 状态有明确展示区分**（进度时间线展示「预约中」标注），用户可感知。

| 功能 | 是否引入新状态码 | 说明 |
|------|:---------------:|------|
| 暂存（草稿） | **否** | 存于表单层，提交前与状态机完全隔离 |
| 预约发送 | **内部逻辑需要，对外状态码由开发决定** | B-b（预约等待中）需在列表展示层做区分 |

## 五、数据统计口径

### 「处理中金额」统计范围

账户顶部「处理中金额」的统计口径：

| 状态 | 是否计入「处理中金额」 | 原因 |
|------|:------------------:|------|
| 草稿 | ❌ | 尚未提交，余额未冻结 |
| 状态 A（等待合同签署） | ✅ | 已提交，进入处理流程 |
| 状态 A1（审核中） | ✅ | 已提交，进入处理流程 |
| 状态 B-b（预约等待中） | ✅ | 审核已通过，余额已冻结 |
| 状态 B-c（等待支付通道处理） | ✅ | 审核已通过，余额已冻结，等待进入转账环节 |

## 六、异常处理

| 异常场景 | 处理方式 |
|---------|---------|
| 草稿数量已达上限（20 条） | 阻止新建草稿，提示：「草稿数量已达上限，请删除旧草稿后再创建」 |
| 预约时间时区不一致 | 以服务器时区为准，UI 选择器旁标注当前时区 |
| 预约触发转账时余额问题 | 不存在此问题。余额检查发生在进入状态 A 之前，审核通过（状态 B-a）时对应金额已从余额中冻结，B-b/B-c 均只是等待和释放已冻结资金，不再做余额校验 |

## 七、验收标准

### 功能一：支出单暂存（草稿）

- [ ] 未选网红频道时，「保存草稿」按钮置灰，悬停显示 tooltip
- [ ] 选择网红频道后，「保存草稿」可点击；点击后弹窗关闭，列表顶部出现草稿条目
- [ ] 关闭新建弹窗时，若已选网红频道，弹出二选保存确认；若未选，直接关闭
- [ ] 「继续编辑」后修改了内容未保存就关闭，弹出三选确认（保存草稿 / 不保存恢复 / 取消不关闭）
- [ ] 草稿仅创建人可见，其他成员登录后看不到
- [ ] 草稿条目支持「继续编辑」（打开弹窗并恢复已填内容）
- [ ] 草稿条目支持「删除」（带二次确认）
- [ ] 草稿提交成功后，草稿记录自动从列表消失
- [ ] 单账号草稿超过 20 条时，阻止新建并给出提示
- [ ] 多条草稿按最后编辑时间倒序排列

### 功能二：预约发送

- [ ] 创建表单底部操作区并列显示：「保存草稿」/「立即发送」/「预约发送」
- [ ] 选中「预约发送」后，日期+小时选择器展开，3 天内时间置灰不可选
- [ ] 时间选择器旁显示平台时区标注
- [ ] 创建成功后，列表时间线第三节点显示「开始转账（预约中）」+ 预约时间
- [ ] B-b 预约等待阶段，操作区显示「修改预约时间」和「转为立即发送」按钮
- [ ] 到达预约时间且支付单已到 B-b，系统自动触发 B-c
- [ ] 预约失败时，广告主收到站内信；消息按原因 A/B 分类，包含三种操作入口
- [ ] 三种失败处理操作（重设时间 / 立即发送 / 作废）均可正常执行
- [ ] 预约失败后重设时间，最早可选时间 = 当前时间 + 72h
- [ ] 预约时间前 48 小时，广告主收到站内信提醒
- [ ] 状态 A 支持修改预约时间和转为立即发送
- [ ] 状态 A1 支持转为立即发送，不支持修改预约时间
