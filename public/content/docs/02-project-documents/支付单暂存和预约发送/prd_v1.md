## 1. 项目概述

### 1.1 背景与目标

**模块入口路径**：合作管理 → 支付管理

**核心痛点**：
- 创建支出单为一次性操作，中途离开即丢失所有填写内容；用户反馈在填写复杂收款信息（银行账号、地址等）时被迫中断后需重头填写
- 支出单提交后一旦审核通过即立即触发转账，无法控制打款时机；品牌方需要与合作网红协商好合同签字时间后，才能在合适的日期发起实际转账

**本次上线范围**（两个功能同期上线）：

| 功能 | 简述 |
|------|------|
| 支出单暂存（草稿） | 允许广告主在创建支出单过程中将未填完的表单保存为草稿，后续可继续编辑或直接提交 |
| 预约发送 | 允许广告主在提交支出单时选择「预约发送」，系统在满足前置条件后自动在指定时间触发转账 |

### 1.2 与现有状态机的关系

**草稿（暂存）**：存储于表单层，**独立于支付单状态机之外**，不占用任何状态码。草稿提交后才正式进入状态机（进入状态 A）。

> ⚠️ 易混淆点：现有状态机中「状态 A」名称为「暂存支付单，等待合同签字完毕」，这是**正式提交后**的状态，与本功能的「草稿」含义不同，请注意区分。

**预约发送**：不引入新的**对外状态码**，但在 B→B1 之间引入内部逻辑状态 **B-b（预约等待中）**，需在列表层对该状态有明确的展示区分。

### 1.3 支付单完整状态机（含预约发送扩展）

```
初始
 └─→ A（暂存支付单，等待合同签字完毕）
      ├─→ A2（广告主在合同签字前取消支付）【终止】
      ├─→ A4（预约发送专属：预约时间到达但合同未完成签署）【可操作异常】
      └─→ A1（合同签字完成，支付审核中）
           ├─→ A3（广告主在审核完成前取消支付）【终止】
           ├─→ A5（预约发送专属：预约时间到达但审核未在截止前完成）【可操作异常】
           ├─→ C（审核未通过）【终止】
           └─→ B（审核通过）
                ├─→ [立即发送] → B-c（开始转账）
                └─→ [预约发送] → B-b（预约等待中）
                     ├─→ [时间到达] → B-c（开始转账）
                     └─→ [用户主动] → B-c（转为立即发送）
                B-c
                └─→ B1（支付通道处理中）
                     ├─→ B2（支付通道处理失败）【终止】
                     └─→ B4（已转入收款行）
                          └─→ D（收款行业务异常，退款中）
                               └─→ D1（退款已到账）【终止】
```

> **A4/A5 实现说明**：A4/A5 为**属性标记（flag）叠加在底层状态 A/A1 之上**，不新增对外状态码；后端通过独立字段（如 `schedule_timeout: "A4"/"A5"`）标记，前端据此差异化展示「预约超时」状态并提供处理操作。**无预约的普通支付单该字段不存在**（而非传 `null`），前端以字段是否存在作为判断依据，避免 `undefined/null/"A4"/"A5"` 四值歧义。
>
> **设计决策说明**：B-b 阶段（审核已通过、余额已冻结）不存在「预约失败」场景——支付单流转到 B 意味着合同已签 + 审核已通过，B-b 的等待是纯时间等待，无任何前置条件可再失败。真实的超时失败发生在 A/A1 阶段，分别对应 A4（合同未完成签署）和 A5（审核未在截止前完成）。

## 2. 全局规则

### 2.1 状态 B 的拆分（预约发送引入）

引入预约发送后，原状态 B（「审核通过，等待支付通道处理」）在内部区分为三个阶段：

| 内部阶段 | 含义 | 预约支付单 | 立即发送支付单 |
|---------|------|-----------|--------------|
| B-a | 审核通过，系统判断是否有预约 | 进入 B-b | 直接进入 B-c |
| **B-b** | **预约等待中**（等待预约时间到达） | 存在此阶段 | 无此阶段 |
| B-c | 等待支付通道处理（即原 B1） | 由 B-b 触发 | 由 B-a 直接触发 |

> **实现说明**：B-a/B-b 为内部逻辑状态，是否新增对外状态码由后端开发决定。Product 要求：**列表层面对 B-b 状态有明确展示区分**（进度时间线展示「预约中」标注），用户可感知。
>
> **预约超时异常（A4/A5）**：原 B 序列不存在预约失败路径。预约时间到达时，若支付单仍在 A/A1，系统标记为 A4/A5（flag 叠加，详见 §1.3 说明），用户可在此状态下选择重新预约、转为立即发送或作废。

### 2.2 「处理中金额」统计口径

账户顶部「处理中金额」的统计范围：

| 状态 | 是否计入 | 原因 |
|------|:--------:|------|
| 草稿 | ❌ | 尚未提交，余额未冻结 |
| 状态 A（等待合同签署） | ✅ | 已提交，进入处理流程 |
| 状态 A4（预约超时，合同未签） | ✅ | 仍在处理流程中，等待用户处理 |
| 状态 A1（审核中） | ✅ | 已提交，进入处理流程 |
| 状态 A5（预约超时，审核未完成） | ✅ | 仍在处理流程中，等待用户处理 |
| 状态 B-b（预约等待中） | ✅ | 审核已通过，余额已冻结 |
| 状态 B-c（等待支付通道处理） | ✅ | 审核已通过，余额已冻结，等待进入转账环节 |

### 2.3 草稿与正式支付单的转换关系

```
草稿（表单层，独立于状态机）
  → 用户点击「立即发送」或「预约发送」提交
  → 正式创建支付单，进入状态 A（等待合同签字）
  → 草稿记录自动删除
```

草稿阶段与状态机完全隔离，提交前对状态机零影响。

**草稿 ID 传递链与自动清除机制**（P2 补充）：

- 「继续编辑」打开创建弹窗时，前端记录当前草稿的 `draft_id`
- 用户点击「立即发送」或「预约发送」提交时，请求携带 `draft_id` 字段
- 后端在正式创建支付单成功后，**联动删除**对应 `draft_id` 的草稿记录，无需前端再单独调用删除接口
- 若提交失败（如校验不通过），草稿记录保留，用户可修改后重新提交
- 从「全新创建」弹窗提交时，请求**不携带 `draft_id` 字段**（字段缺失，而非传空字符串或 `null`），后端以字段是否存在作为判断依据，不执行草稿清除；前端不应传空值以避免后端三值歧义

## 3. 功能一：支出单暂存（草稿）

### 3.1 保存触发条件

#### 最低可保存条件

**网红频道字段已选择**（其他字段可以为空）。

#### 触发方式一：主动保存（按钮点击）

创建表单内显示「保存草稿」按钮：

| 按钮状态 | 触发条件 | 视觉表现 |
|---------|---------|---------|
| 置灰（不可点） | 未选择网红频道 | 按钮置灰，光标 `not-allowed`，悬停显示 tooltip：「请先选择网红频道」 |
| 正常可点 | 已选择网红频道 | 正常线框样式 |
| 加载中 | 点击后等待接口返回 | 按钮内显示 loading 转圈，文字变为「保存中…」，不可重复点击 |

#### 草稿编辑时网红频道锁定

**触发时机**：用户点击「继续编辑」后，打开创建弹窗时。

| 属性 | 规格 |
|------|------|
| 锁定字段 | 「网红频道」选择字段（第一字段） |
| 视觉表现 | 字段变为只读展示态（不可交互），展示已选网红的头像 + 频道名 |
| 交互限制 | 用户无法修改或清除已选网红；字段上无编辑图标、无清除按钮 |
| 兜底操作 | 如需更换网红，须删除此草稿后重新创建 |
| 悬停 tooltip | 「草稿中的网红频道不可修改，如需更换请删除草稿后重新创建」 |

> **设计原因**：草稿与特定网红强绑定（收款信息、币种、金额均与网红相关），更换网红等同于创建全新支出单。锁定网红可防止用户意外修改导致数据不一致。

#### 触发方式二：关闭弹窗时系统提示

用户点击弹窗关闭（×）或「取消」按钮时，根据场景给出不同提示：

**场景一：全新创建时关闭（二选）**

| 子场景 | 行为 |
|--------|------|
| 已选择网红频道（满足最低条件） | 弹出「是否保存草稿？」确认弹窗 |
| 未选择网红频道（不满足最低条件） | 直接关闭弹窗，无任何提示 |

「是否保存草稿？」确认弹窗内容：

| 元素 | 内容 |
|------|------|
| 标题 | 是否保存草稿？ |
| 正文 | 您可以稍后在列表中继续编辑此草稿。 |
| 按钮（主操作） | 保存草稿 |
| 按钮（次要操作） | 不保存，直接关闭 |

**场景二：继续编辑已有草稿后关闭（三选）**

| 子场景 | 行为 |
|--------|------|
| 有修改未保存 | 弹出「尚未保存修改」确认弹窗 |
| 无修改（内容与上次保存完全一致） | 直接关闭弹窗，无提示 |

「尚未保存修改」确认弹窗内容：

| 元素 | 内容 |
|------|------|
| 标题 | 尚未保存修改 |
| 正文 | 您的修改尚未保存，是否保存？ |
| 按钮（主操作） | 保存草稿 |
| 按钮（次要操作） | 不保存，恢复到上次保存 |
| 按钮（兜底操作） | 取消 |

> **「取消」按钮语义**：关闭此确认弹窗，**留在创建弹窗内继续编辑**，不执行关闭。
> 与「不保存，恢复到上次保存」的区别：后者关闭弹窗并回滚，前者保持弹窗开启。

> **「有修改」判断基准**：以最后一次保存成功时的表单字段快照为基准，前端做字段值深比较（deep equal）。完全一致则视为「无修改」；用户修改后又改回原值也视为「无修改」。判断在前端本地完成，无需调接口。

### 3.2 草稿存储规则

| 规则项 | 值 |
|--------|-----|
| 存储层 | 后端持久化（可跨设备恢复，非 localStorage） |
| 有效期 | **永不自动过期**，用户手动删除 |
| 单账号数量上限 | **无上限**（当前阶段不设限制，待上线后根据实际数据决定是否引入保护策略） |
| 可见范围 | **仅创建人**（同账号其他成员不可见） |
| 保存内容 | 仅保存表单填写字段；**不含发送时间**（发送时间属于提交操作，与保存草稿互斥）；**不含文件上传类字段**（当前表单暂无附件字段，如未来新增需单独约定） |

### 3.3 草稿在列表中的展示

#### 列表位置与排序

- 草稿条目**置顶展示**，排在所有正式支付单之前
- 多条草稿之间，**按最后编辑时间倒序排列**（最近编辑的排最前）

#### 状态标签

| 属性 | 规格 |
|------|------|
| 标签文本 | 草稿 |
| 颜色 | 灰色系（`#F8F8F8` 背景，`#6B7280` 文字） |
| 圆角 | `radius-sm` = 4px |
| 无图标前缀 | 与「进行中」「已取消」同等级别 |

> 参考：「进行中」为蓝色，「已取消」为橙色，「草稿」为灰色（语义：未激活/未正式提交）。

#### 列表行展示字段

| 字段 | 草稿行的值 |
|------|-----------|
| 网红头像 + 频道名 | 已填，必有 |
| 支出单 ID | **不显示**（草稿尚未生成正式 ID） |
| 创建时间 | 显示草稿的「最后编辑时间」（格式：「2 小时前编辑」；悬停 tooltip 显示完整时间戳，如 `2026-03-03 14:30`） |
| 币种 | 已填则显示，未填则「—」 |
| 收款人将收到金额 | 已填则显示，未填则「—」 |
| 您共支出金额 | 已填则显示，未填则「—」 |
| 任务名称 | 已填则显示，未填则「—」 |
| 状态标签 | 「草稿」（灰色） |
| 进度时间线 | **不显示**（时间线从正式提交后才出现） |
| 操作按钮 | 「继续编辑」/「删除」 |

#### 筛选支持

「支出状态」筛选下拉完整选项及排列顺序（P5 补充）：

| 顺序 | 选项文案 | 说明 |
|------|---------|------|
| 1 | 全部 | 默认选中，展示所有状态 |
| 2 | 进行中 | 状态 A/A1/B-b/B-c/B1/B4 |
| 3 | 草稿 | 新增：单独筛选草稿条目 |
| 4 | 预约中 | 单独筛选 B-b 预约等待中 |
| 5 | 预约超时 | 新增：单独筛选 A4/A5 预约超时异常 |
| 6 | 已取消 | 状态 A2/A3/C |
| 7 | 已完成 | 状态 D1/B2（转账结束态） |

空状态文案：筛选「草稿」但账号无任何草稿时，显示「暂无草稿，点击「创建营销支出单」开始创建」。

### 3.4 草稿操作规格

#### 继续编辑

| 属性 | 规格 |
|------|------|
| 按钮样式 | 图标按钮（✏ 编辑图标） |
| 触发条件 | 随时可用 |
| 行为 | 打开创建弹窗，恢复所有已保存字段内容；用户可再次保存草稿或直接提交 |

#### 删除

| 属性 | 规格 |
|------|------|
| 按钮样式 | 图标按钮（🗑 删除图标） |
| 触发条件 | 随时可用 |
| 行为 | 弹出删除确认弹窗 |

删除确认弹窗内容：

| 元素 | 内容 |
|------|------|
| 标题 | 确认删除草稿？ |
| 正文 | 删除后无法恢复，确认删除？ |
| 按钮（危险主操作） | 确认删除（危险红，`#EF4444` 待确认⚠️） |
| 按钮（次要操作） | 取消 |

#### 草稿自动清除

草稿提交成功后，草稿记录**自动从列表消失**，无需用户操作。仅在提交成功时触发。

#### 多设备并发删除异常处理（P1 补充）

**场景**：设备 A 正在编辑草稿期间，设备 B（同账号另一会话）删除了同一条草稿；设备 A 随后点击「保存草稿」时，后端发现该 `draft_id` 已不存在。

**处理规则**：

| 场景 | 后端行为 | 前端展示 |
|------|---------|---------|
| 保存时草稿已被删除（`draft_id` 不存在） | 返回特定错误码（如 `DRAFT_NOT_FOUND`） | 弹出提示弹窗：「草稿已被删除，是否另存为新草稿？」 |
| 用户选择「另存为新草稿」 | 后端创建新草稿记录（不带 `draft_id`） | 弹窗关闭，列表顶部出现新草稿条目；Toast 提示「草稿保存成功」 |
| 用户选择「取消」 | 无操作 | 弹窗关闭，用户继续留在编辑弹窗内 |

> 提示弹窗按钮：**另存为新草稿**（主操作，橙色）/ **取消**（次要操作）

## 4. 功能二：预约发送

### 4.1 创建表单底部操作区

> **架构说明**：「发送时间」不是表单填写字段，而是**提交操作的一部分**。「保存草稿」与「立即发送」/「预约发送」互斥，草稿中不保存发送时间。

创建支出单表单底部，三个操作按主次分层排列：

| 操作 | 按钮样式 | 位置 | 性质 |
|------|---------|------|------|
| 保存草稿 | 线框次要按钮（灰色边框） | 左侧 | 辅助操作 |
| 立即发送 | 填充主要按钮（主色橙 `#FF6B35`） | 右侧靠内 | 主操作（默认） |
| 预约发送 | 线框主要按钮（主色橙边框） | 最右侧 | 主操作变体 |

**说明**：「立即发送」为默认主操作，视觉权重最高；「预约发送」为次要主操作；「保存草稿」为辅助操作，视觉权重最低。

### 4.2 「预约发送」时间选择器

用户点击「预约发送」后，按钮变为选中态（高亮边框），在操作按钮行下方展开时间选择区域：

#### 快速选择

| 选项 | 行为 |
|------|------|
| +3天 | 自动填入「当前时间 + 72 小时」对应的日期，小时取当前小时 |
| +7天 | 自动填入「当前时间 + 7 天」对应的日期，小时取当前小时 |
| +14天 | 自动填入「当前时间 + 14 天」对应的日期，小时取当前小时 |

> 快选后用户仍可手动调整日期/小时。

#### 日期选择（日历面板）

| 规则 | 说明 |
|------|------|
| 最早可选时间 | 创建时间 + 72 小时（3 天后）；不足 3 天的日期置灰不可选 |
| 最晚可选时间 | 无硬性限制（UI 默认展示未来 3 个月内） |
| 默认选中 | 最早可选日期（当前日期 + 3 天） |

#### 小时下拉

- 选项：00、01、02 … 23（24 小时制）
- 若所选日期为最早可选日（恰好 +3 天），则不足 72 小时的小时数**置灰不可选**
  - 示例：当前时间 14:30，+72h 截止为 day+3 的 14:30，故 day+3 的 **00~14 小时**均应置灰（小时精度，向上取整到下一可用整点）
- **前端以小时为粒度**做置灰（选项为整点，用户无法选到分钟）
- **后端最终提交校验以分钟为精度**：若预约时间 < 提交时刻 + 72h（精确分钟），则拒绝提交并返回错误提示「预约时间不足 72 小时，请重新选择」；此兜底校验用于应对用户在表单填写期间时间自然流逝导致前端置灰判断失效的边界情况

#### 时区标注

固定展示于时间选择器下方：「时区：UTC+8（北京时间）」

如未来平台需支持多时区，此处保留扩展空间。

#### 24 小时提示文案（固定，始终可见）

> ⏰ 请在预约时间的 24 小时前完成网红签合同，否则审核可能无法完成，导致预约失败。

|------|------|
| 标题 | 确认转为立即发送？ |
| 正文 | 转换后，审核通过时将立即发起转账，预约时间不再生效。 |

> **A1 状态下的执行时序**：用户确认后，后端立即将该支付单标记为「立即发送」（清除预约 flag）。正在进行中的审核流程不受影响，继续推进。审核通过后，系统检测到该支付单已标记为立即发送，直接进入 B-c，不经过 B-b。

### 4.7 预约修改与取消权限矩阵

| 操作 | 状态 A | 状态 A1 | 状态 B-b | 状态 B-c 及之后 |
|------|:------:|:-------:|:--------:|:--------------:|
| 修改预约时间 | ✅ | ❌ | ✅ | ❌ |
| 取消预约（转为立即发送） | ✅ | ✅ | ✅ | ❌ |

**规则说明**：
- 状态 A1 后**不可**修改预约时间：审核启动后改时间会导致审核结果与预约时间不对应，逻辑混乱
- 状态 A/A1 的「转为立即发送」通过列表操作按钮触发（见 §4.4）
- 状态 B-b 的「转为立即发送」通过列表操作按钮触发（见 §4.3）

### 4.8 自动触发逻辑（成功路径）

系统定时任务轮询，**同时满足以下两个条件**时自动触发 B-b → B-c：

```
AND 条件：
  ① 支付单已到达 B-b（预约等待中）
  ② 当前时间 ≥ 预约发送时间
→ 系统自动执行 B-b→B-c（开始转账）
→ 进度时间线恢复正常流转
```

前端无实时推送，用户刷新页面后可见时间线更新。

**定时任务可靠性**：定时任务失败时（如基础设施异常），应支持自动重试；轮询周期和重试参数由研发根据工程规范自决，PRD 不约定具体 SLA 数值。

### 4.9 预约超时处理（A4/A5 异常路径）

#### 触发条件

预约时间到达时，支付单**仍在状态 A 或 A1**（尚未通过审核进入 B 序列）。

| 异常状态 | 触发条件 | 底层状态 |
|---------|---------|---------|
| A4（预约超时 — 合同未签） | 预约时间到达时，支付单仍在状态 A（合同签署未完成） | flag 叠加在 A 之上 |
| A5（预约超时 — 审核未完成） | 预约时间到达时，支付单仍在状态 A1（合同已签但审核未在截止前完成） | flag 叠加在 A1 之上 |

#### 系统动作

1. 标记该支付单为 A4 或 A5（后端独立字段标记，前端展示「预约超时」状态标签）
2. 立即向广告主发送站内信通知（具体模板见 §4.11）

**A4/A5 标记可靠性**：标记动作由定时任务执行，任务失败时应支持自动重试；轮询周期和重试参数由研发根据工程规范自决，PRD 不约定具体 SLA 数值。

#### A4/A5 状态下用户可选的三种处理方式

| 操作 | 说明 | 入口 | 执行后状态 |
|------|------|------|-----------|
| 重新预约 | 设置新预约时间；**最早可选时间 = 当前操作时刻 + 72 小时**；等新时间到达时再次检测 A/A1 状态；复用「修改预约时间」弹窗 UI（与状态 A/B-b 下相同） | 列表行操作 / 站内信 | 清除 A4/A5 flag，继续在 A/A1 流转；列表时间线**即时更新**为新预约时间（接口成功后前端刷新，不保留旧预约时间历史记录）；新预约时间到达时再次判断 |
| 转为立即发送 | 取消预约逻辑，支付单继续正常流转（合同签署 → 审核 → B 序列），不再受预约时间约束 | 列表行操作 / 站内信 | 清除 A4/A5 flag，恢复正常流转 |
| 作废 | 支付单进入取消态（等同于 A2/A3），不可恢复 | 列表行操作 / 站内信 | 终止 |

**作废确认弹窗**：

| 元素 | 内容 |
|------|------|
| 标题 | 确认作废此支付单？ |
| 正文 | 作废后支付单无法恢复，余额将退还至账户余额。 |
| 按钮（危险主操作） | 确认作废 |
| 按钮（次要操作） | 取消 |

**A4/A5 状态的列表行操作按钮**：

```
👁查看 | 重新预约 | 转为立即发送 | 作废
```

## 7. 验收标准

### 7.1 功能一：支出单暂存（草稿）

- [ ] 未选网红频道时，「保存草稿」按钮置灰，悬停显示 tooltip「请先选择网红频道」
- [ ] 选择网红频道后，「保存草稿」可点击；点击后弹窗关闭，列表顶部出现草稿条目，Toast 提示「草稿保存成功」
- [ ] 关闭新建弹窗时，若已选网红频道，弹出「是否保存草稿？」二选弹窗；若未选，直接关闭
- [ ] 「继续编辑」草稿后有修改未保存就关闭，弹出三选确认（保存草稿 / 不保存恢复 / 取消不关闭）
- [ ] 草稿仅创建人可见，其他成员登录后看不到
- [ ] 草稿条目支持「继续编辑」（打开弹窗并恢复已填内容）
- [ ] 草稿条目支持「删除」（带「确认删除草稿？」二次确认弹窗）
- [ ] 草稿提交成功后，草稿记录自动从列表消失
- [ ] 多条草稿按最后编辑时间倒序排列，置顶于正式支付单之前
- [ ] 「继续编辑」草稿时，弹窗内「网红频道」字段为只读展示态，不可修改
- [ ] 不同设备登录同一账号，草稿条目可正常显示（验证后端持久化而非 localStorage）

### 7.2 功能二：预约发送

- [ ] 创建表单底部操作区并列显示三按钮：「保存草稿」（左，灰边框）/「立即发送」（右，主色橙填充）/「预约发送」（最右，主色橙边框）
- [ ] 点击「预约发送」后，按钮变为选中态，下方展开时间选择区域（快选 + 日历 + 小时下拉 + 时区标注 + 24h 提示）
- [ ] 3 天内日期在日历中置灰不可选；最早可选日的不足 72h 小时数在小时下拉中置灰
- [ ] 时间选择器下方固定显示时区标注：「时区：UTC+8（北京时间）」
- [ ] 创建预约发送成功后，列表时间线第三节点显示「开始转账（预约中）」+ 预约时间，旁边显示 ℹ 图标（悬停显示倒计时）
- [ ] B-b 预约等待阶段，操作区显示「修改预约时间」和「转为立即发送」按钮
- [ ] 到达预约时间且支付单已到 B-b，系统自动触发 B-c（刷新后可见时间线更新）
- [ ] 预约超时时（支付单仍在 A/A1），广告主收到站内信；消息按 A4/A5 分类，包含三种操作入口（重新预约 / 转为立即发送 / 作废支付单）
- [ ] 预约超时后「重新预约」，最早可选时间 = 当前操作时刻 + 72h（非原创建时间）
- [ ] 预约时间前 48 小时，广告主收到站内信提醒，内容包含「查看支付单」入口
- [ ] 状态 A：显示「修改预约时间」和「转为立即发送」；点击「修改预约时间」弹出修改弹窗，最早可选 = 操作时刻 + 72h
- [ ] 状态 A1：**不显示**「修改预约时间」；显示「转为立即发送」
- [ ] 预约超时（A4/A5）时，列表行显示「预约超时」状态标签，操作按钮区显示「重新预约 / 转为立即发送 / 作废」
- [ ] A4/A5 状态下点击「转为立即发送」确认后，「预约超时」标签消失，列表行恢复正常操作按钮

### 7.3 异常场景处理

| 异常场景 | 预期行为 |
|---------|---------|
| 多设备并发：设备 B 删除草稿后，设备 A 尝试保存同一草稿 | 后端返回 `DRAFT_NOT_FOUND`；前端弹出提示弹窗「草稿已被删除，是否另存为新草稿？」；用户可选择「另存为新草稿」（创建新记录）或「取消」（留在编辑弹窗） |
| 预约时间不足 72 小时（后端最终校验） | 后端拒绝提交；Toast 提示「预约时间不足 72 小时，请重新选择」；表单保持开启，用户重新选择时间 |
| 预约时间时区不一致 | 以服务器时区（UTC+8）为准，UI 选择器旁标注当前时区 |
| 预约触发转账时余额问题 | 不存在此问题。余额检查发生在进入状态 A 之前，审核通过（状态 B-a）时对应金额已从余额中冻结，B-b/B-c 只是等待和释放已冻结资金，不再做余额校验 |
| 预约时间到达但用户未刷新页面 | 用户停留在列表页期间预约时间到达（B-b→B-c 已触发），前端**不实时推送**，页面仍显示 B-b 旧状态；用户手动刷新页面后，列表时间线更新为 B-c 状态（此为预期行为，验收时须确认刷新后状态正确更新，而非页面始终停留在 B-b） |
