## 1. 项目概述

### 1.1 背景与目标

**模块入口路径**：合作管理 → 支付管理 → 收支明细

**核心痛点**：
- 创建支出单为一次性操作，中途离开即丢失所有填写内容；用户反馈在填写复杂收款信息（银行账号、地址等）时被迫中断后需重头填写
- 支出单提交后一旦审核通过即立即触发转账，无法控制打款时机；品牌方需要与合作网红协商好合同签字时间后，才能在合适的日期发起实际转账

**本次上线范围**（两个功能同期上线）：

| 功能 | 简述 |
|------|------|
| 支出单暂存（草稿） | 允许广告主在创建支出单过程中将未填完的表单保存为草稿，后续可继续编辑或直接提交 |
| 预约发送 | 允许广告主在提交支出单时选择「预约发送」，系统在满足前置条件后自动在指定时间触发转账 |

### 1.2 与现有状态机的关系

**草稿（暂存）**：存储于表单层，**独立于支付单状态机之外**，不占用任何状态码。草稿提交后才正式进入状态机（进入状态 A）。

> ⚠️ 易混淆点：现有状态机中「状态 A」名称为「暂存支付单，等待合同签字完毕」，这是**正式提交后**的状态，与本功能的「草稿」含义不同，请注意区分。

**预约发送**：不引入新的**对外状态码**，但在 B→B1 之间引入内部逻辑状态 **B-b（预约等待中）**，需在列表层对该状态有明确的展示区分。

### 1.3 支付单完整状态机（含预约发送扩展）

```
初始
 └─→ A（暂存支付单，等待合同签字完毕）
      ├─→ A2（广告主在合同签字前取消支付）【终止】
      └─→ A1（合同签字完成，支付审核中）
           ├─→ A3（广告主在审核完成前取消支付）【终止】
           ├─→ C（审核未通过）【终止】
           └─→ B（审核通过）
                ├─→ [立即发送] → B-c（开始转账）
                └─→ [预约发送] → B-b（预约等待中）
                     ├─→ [时间到达 + 条件满足] → B-c（开始转账）
                     └─→ [时间到达 + 条件不满足] → B-b-failed（预约失败）
                          ├─→ 用户「重新预约」→ 重回 B-b
                          ├─→ 用户「转为立即发送」→ 恢复正常流转（等待 B-c）
                          └─→ 用户「作废」→ 终止态
                B-c
                └─→ B1（支付通道处理中）
                     ├─→ B2（支付通道处理失败）【终止】
                     └─→ B4（已转入收款行）
                          └─→ D（收款行业务异常，支付失败，退款中）
                               └─→ D1（退款已到账）【终止】
```

> **B-b-failed 实现说明**：B-b-failed 为**属性标记（flag）叠加在 B-b 底层状态之上**，不新增对外状态码；后端通过独立字段（如 `schedule_failed: true`）与正常 B-b 区分，前端据此差异化展示。

## 2. 全局规则

### 2.1 状态 B 的拆分（预约发送引入）

引入预约发送后，原状态 B（「审核通过，等待支付通道处理」）在内部区分为三个阶段：

| 内部阶段 | 含义 | 预约支付单 | 立即发送支付单 |
|---------|------|-----------|--------------|
| B-a | 审核通过，系统判断是否有预约 | 进入 B-b | 直接进入 B-c |
| **B-b** | **预约等待中**（等待预约时间到达） | 存在此阶段 | 无此阶段 |
| **B-b-failed** | **预约失败**（时间到达但前置条件未满足） | 存在此阶段 | 无此阶段 |
| B-c | 等待支付通道处理（即原 B1） | 由 B-b 触发 | 由 B-a 直接触发 |

> **实现说明**：B-a/B-b 为内部逻辑状态，是否新增对外状态码由后端开发决定。Product 要求：**列表层面对 B-b 状态有明确展示区分**（进度时间线展示「预约中」标注），用户可感知。B-b-failed 为 flag 叠加在 B-b 之上（详见 §1.3 说明）。

### 2.2 「处理中金额」统计口径

账户顶部「处理中金额」的统计范围：

| 状态 | 是否计入 | 原因 |
|------|:--------:|------|
| 草稿 | ❌ | 尚未提交，余额未冻结 |
| 状态 A（等待合同签署） | ✅ | 已提交，进入处理流程 |
| 状态 A1（审核中） | ✅ | 已提交，进入处理流程 |
| 状态 B-b（预约等待中） | ✅ | 审核已通过，余额已冻结 |
| 状态 B-b-failed（预约失败） | ✅ | 余额仍处于冻结状态，等待用户处理（重新预约 / 转立即发送 / 作废） |
| 状态 B-c（等待支付通道处理） | ✅ | 审核已通过，余额已冻结，等待进入转账环节 |

### 2.3 草稿与正式支付单的转换关系

```
草稿（表单层，独立于状态机）
  → 用户点击「立即发送」或「预约发送」提交
  → 正式创建支付单，进入状态 A（等待合同签字）
  → 草稿记录自动删除
```

草稿阶段与状态机完全隔离，提交前对状态机零影响。

## 3. 功能一：支出单暂存（草稿）

### 3.1 保存触发条件

#### 最低可保存条件

**网红频道字段已选择**（其他字段可以为空）。

#### 触发方式一：主动保存（按钮点击）

创建表单内显示「保存草稿」按钮：

| 按钮状态 | 触发条件 | 视觉表现 |
|---------|---------|---------|
| 置灰（不可点） | 未选择网红频道 | 按钮置灰，光标 `not-allowed`，悬停显示 tooltip：「请先选择网红频道」 |
| 正常可点 | 已选择网红频道 | 正常线框样式 |
| 加载中 | 点击后等待接口返回 | 按钮内显示 loading 转圈，文字变为「保存中…」，不可重复点击 |

#### 触发方式二：关闭弹窗时系统提示

用户点击弹窗关闭（×）或「取消」按钮时，根据场景给出不同提示：

**场景一：全新创建时关闭（二选）**

| 子场景 | 行为 |
|--------|------|
| 已选择网红频道（满足最低条件） | 弹出「是否保存草稿？」确认弹窗 |
| 未选择网红频道（不满足最低条件） | 直接关闭弹窗，无任何提示 |

「是否保存草稿？」确认弹窗内容：

| 元素 | 内容 |
|------|------|
| 标题 | 是否保存草稿？ |
| 正文 | 您可以稍后在列表中继续编辑此草稿。 |
| 按钮（主操作） | 保存草稿 |
| 按钮（次要操作） | 不保存，直接关闭 |

**场景二：继续编辑已有草稿后关闭（三选）**

| 子场景 | 行为 |
|--------|------|
| 有修改未保存 | 弹出「尚未保存修改」确认弹窗 |
| 无修改（内容与上次保存完全一致） | 直接关闭弹窗，无提示 |

「尚未保存修改」确认弹窗内容：

| 元素 | 内容 |
|------|------|
| 标题 | 尚未保存修改 |
| 正文 | 您的修改尚未保存，是否保存？ |
| 按钮（主操作） | 保存草稿 |
| 按钮（次要操作） | 不保存，恢复到上次保存 |
| 按钮（兜底操作） | 取消 |

> **「取消」按钮语义**：关闭此确认弹窗，**留在创建弹窗内继续编辑**，不执行关闭。
> 与「不保存，恢复到上次保存」的区别：后者关闭弹窗并回滚，前者保持弹窗开启。

> **「有修改」判断基准**：以最后一次保存成功时的表单字段快照为基准，前端做字段值深比较（deep equal）。完全一致则视为「无修改」；用户修改后又改回原值也视为「无修改」。判断在前端本地完成，无需调接口。

### 3.2 草稿存储规则

| 规则项 | 值 |
|--------|-----|
| 存储层 | 后端持久化（可跨设备恢复，非 localStorage） |
| 有效期 | **永不自动过期**，用户手动删除 |
| 单账号数量上限 | **20 条**（超出时阻止新建，提示清理） |
| 可见范围 | **仅创建人**（同账号其他成员不可见） |
| 保存内容 | 仅保存表单填写字段；**不含发送时间**（发送时间属于提交操作，与保存草稿互斥）；**不含文件上传类字段**（当前表单暂无附件字段，如未来新增需单独约定） |

### 3.3 草稿在列表中的展示

#### 列表位置与排序

- 草稿条目**置顶展示**，排在所有正式支付单之前
- 多条草稿之间，**按最后编辑时间倒序排列**（最近编辑的排最前）

#### 状态标签

| 属性 | 规格 |
|------|------|
| 标签文本 | 草稿 |
| 颜色 | 灰色系（`#F8F8F8` 背景，`#6B7280` 文字） |
| 圆角 | `radius-sm` = 4px |
| 无图标前缀 | 与「进行中」「已取消」同等级别 |

> 参考：「进行中」为蓝色，「已取消」为橙色，「草稿」为灰色（语义：未激活/未正式提交）。

#### 列表行展示字段

| 字段 | 草稿行的值 |
|------|-----------|
| 网红头像 + 频道名 | 已填，必有 |
| 支出单 ID | **不显示**（草稿尚未生成正式 ID） |
| 创建时间 | 显示草稿的「最后编辑时间」（格式：「2 小时前编辑」；悬停 tooltip 显示完整时间戳，如 `2026-03-03 14:30`） |
| 币种 | 已填则显示，未填则「—」 |
| 收款人将收到金额 | 已填则显示，未填则「—」 |
| 您共支出金额 | 已填则显示，未填则「—」 |
| 任务名称 | 已填则显示，未填则「—」 |
| 状态标签 | 「草稿」（灰色） |
| 进度时间线 | **不显示**（时间线从正式提交后才出现） |
| 操作按钮 | 「继续编辑」/「删除」 |

#### 筛选支持

现有「支出状态」筛选下拉新增以下选项：
- **「草稿」**：支持单独筛选草稿条目
- **「预约失败」**：支持单独筛选处于预约失败状态的支付单

空状态文案：筛选「草稿」但账号无任何草稿时，显示「暂无草稿，点击「创建营销支出单」开始创建」。

### 3.4 草稿操作规格

#### 继续编辑

| 属性 | 规格 |
|------|------|
| 按钮样式 | 图标按钮（✏ 编辑图标） |
| 触发条件 | 随时可用 |
| 行为 | 打开创建弹窗，恢复所有已保存字段内容；用户可再次保存草稿或直接提交 |

#### 删除

| 属性 | 规格 |
|------|------|
| 按钮样式 | 图标按钮（🗑 删除图标） |
| 触发条件 | 随时可用 |
| 行为 | 弹出删除确认弹窗 |

删除确认弹窗内容：

| 元素 | 内容 |
|------|------|
| 标题 | 确认删除草稿？ |
| 正文 | 删除后无法恢复，确认删除？ |
| 按钮（危险主操作） | 确认删除（危险红，`#EF4444` 待确认⚠️） |
| 按钮（次要操作） | 取消 |

#### 草稿自动清除

草稿提交成功后，草稿记录**自动从列表消失**，无需用户操作。仅在提交成功时触发。

### 3.5 草稿数量上限交互

| 场景 | 交互 |
|------|------|
| 未达上限 | 「创建营销支出单」按钮正常可用 |
| 已达上限（20 条）时点击创建 | 按钮外观**保持正常**（不置灰）；点击后弹出 Toast 警告 |

Toast 内容：「草稿数量已达上限（20条），请删除旧草稿后再创建」（橙色警告，持续 4 秒）

## 4. 功能二：预约发送

### 4.1 创建表单底部操作区

> **架构说明**：「发送时间」不是表单填写字段，而是**提交操作的一部分**。「保存草稿」与「立即发送」/「预约发送」互斥，草稿中不保存发送时间。

创建支出单表单底部，三个操作按主次分层排列：

| 操作 | 按钮样式 | 位置 | 性质 |
|------|---------|------|------|
| 保存草稿 | 线框次要按钮（灰色边框） | 左侧 | 辅助操作 |
| 立即发送 | 填充主要按钮（主色橙 `#FF6B35`） | 右侧靠内 | 主操作（默认） |
| 预约发送 | 线框主要按钮（主色橙边框） | 最右侧 | 主操作变体 |

**说明**：「立即发送」为默认主操作，视觉权重最高；「预约发送」为次要主操作；「保存草稿」为辅助操作，视觉权重最低。

### 4.2 「预约发送」时间选择器

用户点击「预约发送」后，按钮变为选中态（高亮边框），在操作按钮行下方展开时间选择区域：

#### 快速选择

| 选项 | 行为 |
|------|------|
| +3天 | 自动填入「当前时间 + 72 小时」对应的日期，小时取当前小时 |
| +7天 | 自动填入「当前时间 + 7 天」对应的日期，小时取当前小时 |
| +14天 | 自动填入「当前时间 + 14 天」对应的日期，小时取当前小时 |

> 快选后用户仍可手动调整日期/小时。

#### 日期选择（日历面板）

| 规则 | 说明 |
|------|------|
| 最早可选时间 | 创建时间 + 72 小时（3 天后）；不足 3 天的日期置灰不可选 |
| 最晚可选时间 | 无硬性限制（UI 默认展示未来 3 个月内） |
| 默认选中 | 最早可选日期（当前日期 + 3 天） |

#### 小时下拉

- 选项：00、01、02 … 23（24 小时制）
- 若所选日期为最早可选日（恰好 +3 天），则不足 72 小时的小时数**置灰不可选**
  - 示例：当前时间 14:30，最早可选日期为 3 天后，则该日期下 00~13 小时置灰

#### 时区标注

固定展示于时间选择器下方：「时区：UTC+8（北京时间）」

如未来平台需支持多时区，此处保留扩展空间。

#### 24 小时提示文案（固定，始终可见）

> ⏰ 请在预约时间的 24 小时前完成网红签合同，否则审核可能无法完成，导致预约失败。

### 4.3 列表展示 — B-b 预约等待阶段

#### 进度时间线节点

支付单处于 B-b（预约等待中）时，进度时间线第三节点：

| 节点 | 样式 | 内容 |
|------|------|------|
| 第一节点 | 橙色实心 ● | 支出单已创建 — `{创建时间}` |
| 第二节点 | 橙色实心 ● | 平台审中 — `{审核通过时间}` |
| 第三节点 | 橙色空心 ○ | 开始转账（预约中）— 预约于 `{预约时间（UTC+8）}` |

#### 倒计时气泡（ℹ 图标）

- 在「开始转账（预约中）」节点旁放置一个信息图标（ℹ）
- 鼠标悬停时弹出气泡显示：「还有 **X天X小时** 后自动发送」
- 气泡为只读信息，无操作按钮

#### B-b 阶段列表行操作按钮

```
👁查看 | 修改预约时间 | 转为立即发送 | 取消
```

| 操作 | 点击行为 |
|------|---------|
| 修改预约时间 | 打开「修改预约时间」弹窗 |
| 转为立即发送 | 弹出「确认转为立即发送？」确认弹窗 |
| 取消 | 复用现有「取消支付单」确认弹窗 |

### 4.4 状态 A/A1 列表操作按钮（预约发送专有）

适用范围：该支付单为预约发送 + 状态为 A 或 A1。

**状态 A 操作区**：

```
👁查看 | ✏编辑 | 修改预约时间 | 转为立即发送 | 取消
```

**状态 A1 操作区**：

```
👁查看 | 转为立即发送 | 取消
```

> 状态 A1（审核中）不可修改预约时间（见 §4.7 权限矩阵），因此不显示「修改预约时间」。

### 4.5 修改预约时间弹窗

触发：状态 A 或 B-b 下，点击「修改预约时间」。

| 元素 | 内容 |
|------|------|
| 标题 | 修改预约时间 |
| 当前预约时间 | 只读展示，如「当前预约时间：2026-03-10 14:00（UTC+8）」 |
| 新预约时间 | 与创建时一致（快选 +3/7/14天 + 日历 + 小时下拉） |
| 固定提示 | ⏰ 请确认合同已安排，避免预约失败。 |
| 按钮（主操作） | 确认修改 |
| 按钮（次要操作） | 取消 |

最早可选时间：
- 状态 A：当前操作时刻 + 72h
- 状态 B-b：当前操作时刻 + 72h

### 4.6 「转为立即发送」二次确认弹窗

触发：状态 A/A1/B-b 下，点击「转为立即发送」。

| 元素 | 内容 |
|------|------|
| 标题 | 确认转为立即发送？ |
| 正文 | 转换后，审核通过时将立即发起转账，预约时间不再生效。 |
| 按钮（主操作） | 确认 |
| 按钮（次要操作） | 取消 |

确认后：取消预约，支付单按立即发送逻辑继续流转；列表操作按钮区移除「修改预约时间」/「转为立即发送」。

> **A1 状态下的执行时序**：用户确认后，后端立即将该支付单标记为「立即发送」（清除预约 flag）。正在进行中的审核流程不受影响，继续推进。审核通过后，系统检测到该支付单已标记为立即发送，直接进入 B-c，不经过 B-b。

### 4.7 预约修改与取消权限矩阵

| 操作 | 状态 A | 状态 A1 | 状态 B-b | 状态 B-c 及之后 |
|------|:------:|:-------:|:--------:|:--------------:|
| 修改预约时间 | ✅ | ❌ | ✅ | ❌ |
| 取消预约（转为立即发送） | ✅ | ✅ | ✅ | ❌ |

**规则说明**：
- 状态 A1 后**不可**修改预约时间：审核启动后改时间会导致审核结果与预约时间不对应，逻辑混乱
- 状态 A/A1 的「转为立即发送」通过列表操作按钮触发（见 §4.4）
- 状态 B-b 的「转为立即发送」通过列表操作按钮触发（见 §4.3）

### 4.8 自动触发逻辑（成功路径）

系统定时任务轮询，**同时满足以下两个条件**时自动触发 B-b → B-c：

```
AND 条件：
  ① 支付单已到达 B-b（预约等待中）
  ② 当前时间 ≥ 预约发送时间
→ 系统自动执行 B-b→B-c（开始转账）
→ 进度时间线恢复正常流转
```

前端无实时推送，用户刷新页面后可见时间线更新。

### 4.9 预约失败处理（失败路径）

#### 触发条件

预约时间到达时，支付单**尚未到达 B-b**（合同未签或审核未通过）。

#### 失败原因分类

| 原因类型 | 触发条件 |
|---------|---------|
| 原因 A：合同未完成签署 | 预约时间到达时，支付单仍在状态 A（合同签署未完成） |
| 原因 B：审核未在截止前通过 | 预约时间到达时，支付单仍在状态 A1（合同已签但审核未通过） |

#### 系统动作

1. 标记该支付单为「预约失败」
2. 立即向广告主发送站内信通知（具体模板见 §4.11）

#### 失败后用户可选的三种处理方式

| 操作 | 说明 | 入口 | 执行后状态 |
|------|------|------|-----------|
| 重新预约 | 复用原支付单 ID，仅修改预约时间；**最早可选时间 = 当前操作时刻 + 72 小时** | 列表行操作 / 站内信 | 重新进入预约等待（B-b） |
| 转为立即发送 | 取消预约，一旦前置条件（到达 B-b）满足，立即自动触发 B-c | 列表行操作 / 站内信 | 恢复正常流转 |
| 作废 | 支付单进入取消态（等同于 A2/A3），不可恢复 | 列表行操作 / 站内信 | 终止 |

**作废确认弹窗**：

| 元素 | 内容 |
|------|------|
| 标题 | 确认作废此支付单？ |
| 正文 | 作废后支付单无法恢复，余额将退还至账户余额。 |
| 按钮（危险主操作） | 确认作废 |
| 按钮（次要操作） | 取消 |

**预约失败状态的列表行操作按钮**：

```
👁查看 | 重新预约 | 转为立即发送 | 作废
```

### 4.10 提醒机制

| 提醒类型 | 触发时机 | 渠道 | 内容要点 |
|---------|---------|------|---------|
| 预约前提醒 | 预约时间前 **48 小时**，**且支付单仍处于预约等待状态（A/A1/B-b）**；若支付单已取消、已转为立即发送或已预约失败，不触发 | 站内信 | 提醒预约即将到达，请确认合同已完成签署 |

## 7. 验收标准

### 7.1 功能一：支出单暂存（草稿）

- [ ] 未选网红频道时，「保存草稿」按钮置灰，悬停显示 tooltip「请先选择网红频道」
- [ ] 选择网红频道后，「保存草稿」可点击；点击后弹窗关闭，列表顶部出现草稿条目，Toast 提示「草稿保存成功」
- [ ] 关闭新建弹窗时，若已选网红频道，弹出「是否保存草稿？」二选弹窗；若未选，直接关闭
- [ ] 「继续编辑」草稿后有修改未保存就关闭，弹出三选确认（保存草稿 / 不保存恢复 / 取消不关闭）
- [ ] 草稿仅创建人可见，其他成员登录后看不到
- [ ] 草稿条目支持「继续编辑」（打开弹窗并恢复已填内容）
- [ ] 草稿条目支持「删除」（带「确认删除草稿？」二次确认弹窗）
- [ ] 草稿提交成功后，草稿记录自动从列表消失
- [ ] 单账号草稿超过 20 条时，点击「创建营销支出单」按钮正常，点击后弹出 Toast 警告（不置灰按钮）
- [ ] 多条草稿按最后编辑时间倒序排列，置顶于正式支付单之前
- [ ] 不同设备登录同一账号，草稿条目可正常显示（验证后端持久化而非 localStorage）

### 7.2 功能二：预约发送
