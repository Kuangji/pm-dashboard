# 支付单暂存和预约发送

> 功能模块入口文档 · VDR 工作流 · 适用对象：新接入研发 / 设计人员

## 功能概述

**两个功能同期上线**，在现有支付管理模块基础上新增能力，共用同一个入口（合作管理 → 支付管理）：

| 功能 | 一句话说明 |
|------|-----------|
| 支出单暂存（草稿） | 创建到一半可以保存，下次继续填 |
| 预约发送 | 提交支付单时可以指定「几号几点打款」 |

## 所属模块

- 一级导航：合作管理
- 二级导航：支付管理

## 最关键的 3 个概念

### 1. 草稿 ≠ 状态 A（极易混淆⚠️）

- **草稿**：存在表单层，**不进状态机**，用户自己可删
- **状态 A**（暂存支付单）：正式提交后才有，名字虽然也叫「暂存」但完全不同

> 详见 → `spec_v1.md §一`、`prd_v1.md §1.2`

### 2. 预约发送引入两个新内部状态

```
B-b  预约等待中     审核通过后等待到达预约时间，才触发转账
A4   预约超时-合同未签  预约时间到了但合同还没签完
A5   预约超时-审核未完  预约时间到了但审核还没走完
```

**A4/A5 不是新的对外状态码**，是叠加在 A/A1 上的 flag。

> 完整状态机 → `prd_v1.md §1.3`

### 3. 发送时间不存在草稿里

「保存草稿」**不保存**发送时间，只保存表单字段。
发送时间是「立即发送/预约发送」按钮触发时才提交的。

## 状态流转速查

```
草稿（表单层）
  └─→ 提交 → 状态 A → A1 → B → B-b（预约等待）→ B-c → B1 → B4 → D
                        ↘ A4/A5（预约超时，可操作）
```

**B-b 自动触发条件**（定时任务）：支付单已在 B-b **且** 当前时间 ≥ 预约时间 → 自动转 B-c

> 详见 → `spec_v1.md §三`、`prd_v1.md §3`

## 研发关键规则

| 规则 | 说明 | 出处 |
|------|------|------|
| 草稿存储 | 后端持久化，非 localStorage；跨设备可见 | `spec_v1.md §2.2` |
| 草稿上限 | **无上限** | `spec_v1.md §2.2` |
| 最低保存条件 | 只需选了「网红频道」，其他字段可空 | `spec_v1.md §2.1` |
| 继续编辑时网红频道 | **只读锁定**，不可修改 | `spec_v1.md §2.3` |
| 草稿提交后 | 自动删除草稿记录（无需用户操作） | `spec_v1.md §2.3` |
| 预约最早时间 | 创建操作时刻 + 72h，精确到小时 | `spec_v1.md §3.1` |
| B-b→B-c 触发 | 定时任务轮询，非 Webhook | `spec_v1.md §3.3` |
| A4/A5 触发后 | 立即发站内信通知广告主 | `spec_v1.md §3.4` |
| A4/A5 操作失败 | 前端不区分原因，统一 Toast「操作失败，请重试」 | `design_v1.md §3.6` |
| 重新预约成功后 | 时间线**即时更新**，无需刷新 | `design_v1.md §五` |

## 设计关键规则

| 规则 | 说明 | 出处 |
|------|------|------|
| 草稿置顶 | 草稿排在所有正式支付单上方 | `design_v1.md §2.5` |
| 草稿标签样式 | 灰色背景 `#F3F4F6`，深灰字 `#6B7280` | `design_v1.md §2.4` |
| 预约超时标签 | 红色 `#EF4444`，浅红背景 `#FEF2F2` | `design_v1.md §3.6` |
| B-b 倒计时 | ℹ 图标悬停 Popover 显示，**不内联**在时间线 | `design_v1.md §3.2` |
| A4/A5 操作按钮 | 「重新预约」复用「修改预约时间」弹窗（§3.3）| `design_v1.md §3.6` |
| 全局视觉令牌 | 主色 `#FF6B35`，边框 `#E5E7EB`，弹窗圆角 16px | `design_v1.md §七` |
| 原型参考 | 7 个屏幕，Screen7 = A4/A5 预约超时状态 | `prototype_v1.pen` |
