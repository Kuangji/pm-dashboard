# 频道搜索模式升级 · Design v1

## 一、整体设计原则

- 搜索页外部尽量稳定
- 模式切换尽量清晰
- `标准搜索` 看起来像原能力增强
- `AI搜索` 看起来像独立工作流，但不做成聊天产品

## 二、标准搜索设计

### 2.1 搜索区结构

在标准搜索搜索区内提供轻量形态切换：

- `关键词`
- `自然语言`

建议：

- 放在搜索框上沿或输入区左侧
- 视觉上显式但不喧宾夺主

### 2.2 关键词模式

- 尽量与现有搜索页一致
- 不主动引入 AI 面板感
- `关键词` 继续以胶囊样式留在搜索框内，强调经典搜索里的“成词”编辑心智

### 2.3 自然语言模式

空白输入态表现为：

- 一条更长的自然语言输入区
- `关键词 / 自然语言` 切换继续放在输入区内部
- 输入框默认空白
- 但当前生效条件必须显式可见
- 关键词不再留在输入框内，而是和地区、粉丝量等一起放进 `当前生效条件`
- 明确的提交按钮
- 示例文案偏完整需求表达，可同时覆盖“整套重建”与“增量调整”

提交后页面动作：

- 先进入轻量处理中态
- 再回到现有标准搜索结果页
- 不是进入新结果系统
- 而是把用户拉回现有标准搜索心智

### 2.3.1 自然语言结果页壳层

自然语言成功后，顶部固定为：

1. `紧凑自然语言输入框`
2. `当前生效条件`
3. `折叠历史时间线`

要求：

- 输入框始终可用，用户可以直接输入下一句自然语言继续调参
- 页面重点只放在 `当前生效条件`，不单独做参数 diff 层
- 历史只记录成功搜索，不做聊天记录感
- 点击历史项时只回填原句，不自动立即搜索
- 历史语句后续再次提交时，仍按当前生效条件解释
- 用户若要手动微调结构化条件，应通过输入区内的 `关键词` 切换进入传统搜索态

### 2.3.2 自然语言失败态

- 失败提示贴在输入框下方
- 不切独立错误页
- 若当前已有结果，则旧结果继续保留
- 不自动退回关键词模式
- 不自动跳 AI搜索

### 2.4 标准搜索到 AI搜索 的桥接

标准搜索里必须有一条清楚的桥接线：

- 用户已经有一组可用条件
- 但对当前标准结果不满意
- 可以显式把当前条件带到 `AI搜索` 继续探索

设计要求：

- 该动作应出现在标准结果页里，而不是只在 AI搜索 空白态里出现
- 该动作应挂在 `当前生效条件` 区域里，而不是挂在自然语言输入框旁
- 需要把“将带入当前条件，但不会改写标准搜索状态”说明清楚
- 视觉上是次于主搜索动作、但明显可见的次级 CTA

## 三、AI搜索（智能精选）设计

### 3.1 页面气质

AI搜索应该有独立模式感，但仍是搜索工作台，而不是聊天对话窗。

设计重心：

- 当前候选总池
- 自然语言输入
- 批次结果消费
- 当前 session 集合
- session 历史侧栏
- 下一批动作

### 3.1.1 多 session 工作台

AI 搜索页不再是单栏，而改为：

- 左侧：`session 历史侧栏`
- 右侧：`当前智能精选工作台`

侧栏职责：

- 承接 `新建 session`
- 展示最近使用的未归档 session
- 折叠展示 `已归档`
- 让用户明确知道：每次智能精选都是一个独立过程，而不是覆盖上一次工作

交互要求：

- 再次进入 AI 搜索时，默认落到最近活动的未归档 session
- 切换 session 时，不保留上一个 session 的临时展开态和浮层
- 侧栏本身允许实时 `展开 / 收起`
- 该展开 / 收起只是一层前端 UI 开关，不形成独立场景
- session 默认自动命名，但允许手动改名
- 当前阶段只做归档/恢复，不做永久删除

### 3.2 空白起手态

应包含：

- 当前候选总池卡
- query 输入框
- `开始 AI搜索`
- 次级按钮：`带入当前标准搜索条件`

本轮实施口径：

- `从 0 开始` 时，总池来自全量频道库
- `带入当前标准搜索条件` 时，不再进入占位页，而是进入带预装总池的同一工作台
- 带入的核心是候选总池范围，而不是几条 seed 字段
- 工作台顶部顺序固定为：`总池 -> query -> session 集合 -> 当前批次`
- 若当前尚无未归档 session，右侧显示空白起手态；左侧侧栏仍保留 `新建 session` 与 `已归档` 区

### 3.3 批次执行中

需要同时呈现：

- 当前总池
- 当前 query
- 正在执行的状态条或状态卡
- 结果区 skeleton

避免：

- 页面只剩空白
- 用户不确定是否还在运行

低优先级 TODO：

- 在 demo 中补三种 mock 返回分支
- `召回成功` 进入 `一批结果`
- `低召回` 进入低召回提示与调 query 引导
- `失败或超时` 进入重试或返回编辑 query

### 3.4 批次结果态

结果列表保持专业搜索工具样式：

- 左侧频道身份
- 中部指标
- 右侧最近内容或缩略信息

每张卡新增：

- `采纳`
- `排除`

规则：

- `采纳 / 排除` 单击即生效
- 生效后条目立即从当前主列表移除
- `待定` 不再是正式动作或正式结果桶
- 未处理项只是当前仍滞留在主列表中的结果

### 3.5 顶部摘要

顶部摘要采用：

- `已采纳`
- `已排除`
- 可展开明细与轻量改判

目标：

- 让用户知道本次 session 已经消费了哪些结果
- 但不把页面做成重型侧栏工作台
- `已采纳 / 已排除` 按整个 session 累计
- 当前批次剩余未处理数量不放在顶部，而放在结果区底部
- 从展开集合里可把结果改判为：采纳 / 排除 / 回当前列表未处理

### 3.5.1 侧栏条目

每个 session 条目默认展示：

- session 标题
- 总池来源 badge
- 一条状态优先信息
  - `执行中`
  - `第 N 批 · 剩余 X 条未处理`
  - `自动进入下一批`
  - `已归档`
- 一条弱化 meta：最近更新时间 + 已采纳 / 已排除计数

条目动作：

- hover 显示 `改名`
- hover 显示 `归档`
- 归档区显示 `恢复并打开`
- 当前 active session 需要有稳定高亮

### 3.6 下一批动作

- 当当前批次仍有未处理结果时，下一批是稳定、显式的主按钮
- 放在结果区底部，与“当前还剩多少未处理结果”绑定
- 用户点击后再进入下一轮执行态
- 若本批结果已全部采纳/排除，则给一个短过渡后自动进入下一批
- 用户在仍有未处理结果时主动进入下一批，这些结果被直接放弃，不再保留为第三桶

### 3.7 改 query 重搜

- `改 query` 不是重新开 session
- `改 query` 后，总池不变
- `已采纳 / 已排除` 继续保留
- `已排除` 作为硬过滤，`已采纳` 作为正向偏好信号
- 页面不做 query 历史时间线，始终只强调当前 query
- `改 query` 只影响当前 active session，左侧其他 session 完全不受影响

## 四、关键画面清单

- 标准搜索 / 关键词基线
- 标准搜索 / 自然语言输入
- 标准搜索 / 自然语言解析中
- 标准搜索 / 自然语言回填结果
- 标准搜索 / 带条件跳转 AI搜索
- 标准搜索 / 自然语言解析失败
- AI搜索 / 空白起手
- AI搜索 / 带条件起手
- AI搜索 / 执行中
- AI搜索 / 批次结果
- AI搜索 / 集合展开与改判
- AI搜索 / 修改 query 后重新搜索
