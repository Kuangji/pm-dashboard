# NoxInfluencer 能力拆解

## 主链路

后续做竞品对照、站级评价或需求定位时，建议统一使用这条主链路作为基准：

> **找达人/商品/品牌情报 -> 建档与沉淀 -> 批量触达 -> 实时沟通 -> 监控发布效果 -> 支付结算 -> 配额/权限/平台治理**

这条链路并不意味着所有用户都走完整路径，但它能解释 Nox 为什么同时存在搜索、品牌分析、CRM、邮件、消息、监控、支付、数据用量和 Skills。

---

## 逐模块拆解

### 1. 官网首页与公开入口

- **解决什么业务问题**：让未登录用户快速理解 Nox 提供哪些产品和工具，并进入咨询、定价或工作台。
- **核心对象是什么**：潜在客户、公开工具使用者、内容阅读者。
- **处于链路哪个位置**：链路起点之前的获客层。
- **和前后模块怎么衔接**：把流量导向 `brand-marketing`、`api-service`、`pricing`、`report/blog/help` 和工作台入口。
- **最容易误判的点**：不能把首页上的工具和榜单当成全部产品；首页只是公开入口，不是全站能力总表。

### 2. 海外网红营销云

- **解决什么业务问题**：把 Nox 包装成面向品牌的整体解决方案，而不是单点数据工具。
- **核心对象是什么**：品牌方、代理商、销售线索。
- **处于链路哪个位置**：公开获客层中的核心产品叙事页。
- **和前后模块怎么衔接**：前接官网获客，后接定价、咨询和登录工作台。
- **最容易误判的点**：这页讲的是产品叙事，不代表全部登录后细节都能在公开层看到。

### 3. API 数据服务

- **解决什么业务问题**：面向有系统集成或数据服务诉求的客户，单独售卖数据能力。
- **核心对象是什么**：企业客户、合作方、内部系统或外部平台。
- **处于链路哪个位置**：公开层中的 B2B 数据产品入口。
- **和前后模块怎么衔接**：与品牌营销云并列，不依赖 CRM / 邮件 / 消息这些执行模块。
- **最容易误判的点**：API 服务不是工作台里的一个按钮，而是一条独立商业线。

### 4. 报告 / 博客 / 帮助

- **解决什么业务问题**：教育市场、承接 SEO/内容流量、解释功能与规则。
- **核心对象是什么**：潜在客户、已有客户、需要学习产品的人。
- **处于链路哪个位置**：公开层的内容资产与教育层。
- **和前后模块怎么衔接**：前接官网流量，后接产品转化、功能理解与问题自助解决。
- **最容易误判的点**：这些页面不是“附属内容”，而是 Nox 公开获客和产品教育的重要组成部分。

### 5. 达人搜索工作台

- **解决什么业务问题**：从海量达人中找到值得合作的人，并快速筛选。
- **核心对象是什么**：达人频道、视频、筛选条件、搜索意图。
- **处于链路哪个位置**：主链路最前端的发现入口。
- **和前后模块怎么衔接**：搜索结果通常向收藏夹、CRM、邮件触达、品牌分析等后续动作流动。
- **最容易误判的点**：不能把它仅理解成“搜索页”；它已经包含 AI 推荐关键词、相似网红、其他社媒切换等工作台能力。

### 6. TikTok Shop 搜索

- **解决什么业务问题**：从 TikTok Shop 的商品、店铺、直播维度获取可操作情报。
- **核心对象是什么**：商品、店铺、直播间、类目。
- **处于链路哪个位置**：发现层中的电商/带货分支入口。
- **和前后模块怎么衔接**：既可前接商品或带货机会判断，也可后接商品中心和品牌分析。
- **最容易误判的点**：它不是达人搜索的附属筛选项，而是独立的数据对象体系。

### 7. 品牌榜单与品牌分析

- **解决什么业务问题**：从品牌视角理解竞品推广行为、合作广度、资产沉淀和市场动态。
- **核心对象是什么**：品牌、品牌内容、合作网红、营销资产。
- **处于链路哪个位置**：发现层的“品牌/竞品情报入口”。
- **和前后模块怎么衔接**：可反向指导搜索策略、监控策略和投放判断。
- **最容易误判的点**：它不是内容监控；品牌分析更偏品牌级洞察与策略判断，不是单条内容的持续抓取。

### 8. 商品 / 评论洞察

- **解决什么业务问题**：从商品评论中读取消费者反馈并形成分析报告。
- **核心对象是什么**：商品、评论、评论分析报告。
- **处于链路哪个位置**：发现层与商品运营之间的桥接模块。
- **和前后模块怎么衔接**：可服务于选品、竞品洞察、商品中心和营销素材理解。
- **最容易误判的点**：它不等于 TikTok Shop 搜索；一个偏“站内商品情报”，一个偏“消费者评论洞察”。

### 9. 收藏夹 / 资源夹

- **解决什么业务问题**：把搜索或分析得到的对象先做轻量沉淀，再决定后续动作。
- **核心对象是什么**：资源夹、网红、屏蔽对象、不感兴趣对象。
- **处于链路哪个位置**：发现层与正式关系管理之间的缓冲层。
- **和前后模块怎么衔接**：前接搜索结果，后接 CRM、邮件项目或屏蔽/过滤机制。
- **最容易误判的点**：收藏夹不是 CRM；它更像轻量沉淀和分拣区，而不是完整流程管理区。

### 10. 网红 CRM

- **解决什么业务问题**：把潜在合作对象转成可持续管理的关系资产。
- **核心对象是什么**：网红档案、联系状态、合作状态、标签、成员。
- **处于链路哪个位置**：主链路中的建档与沉淀核心。
- **和前后模块怎么衔接**：前接搜索/收藏夹，后接邮件管理、消息中心、合作流程。
- **最容易误判的点**：CRM 不是消息中心，也不是邮件项目；它是关系主档。

### 11. 邮件管理

- **解决什么业务问题**：用项目化方式对合作对象进行批量触达和邮件运营。
- **核心对象是什么**：邮件项目、模板、收件人、送达与回复统计。
- **处于链路哪个位置**：主链路中的批量触达阶段。
- **和前后模块怎么衔接**：前接 CRM 或收藏对象，后接消息中心和合作状态推进。
- **最容易误判的点**：邮件管理并不等于实时沟通；它更偏外发触达和项目视角的数据统计。

### 12. 消息中心

- **解决什么业务问题**：承接已经建立联系后的日常对话和即时处理。
- **核心对象是什么**：会话、消息状态、待处理事项、AI 回复辅助。
- **处于链路哪个位置**：主链路中的实时沟通阶段。
- **和前后模块怎么衔接**：前接邮件或其他建联动作，后接合作推进、内容监控与支付。
- **最容易误判的点**：消息中心不是 CRM 的一个 tab；它是单独的沟通工作台。

### 13. 智能营销计划

- **解决什么业务问题**：把营销目标转成自动执行或半自动执行的计划。
- **核心对象是什么**：营销计划、目标、AI 执行流程。
- **处于链路哪个位置**：位于执行链路之上的自动化编排层。
- **和前后模块怎么衔接**：理论上可调用搜索、匹配、触达与谈判动作，是更高层的 orchestration。
- **最容易误判的点**：不要把它和 `ai-assistant` 混为一谈；前者是已见到的业务计划页，后者本轮是 blocked page。

### 14. 内容监控 / 自动追踪

- **解决什么业务问题**：持续监控发布内容、追踪上线、分析效果。
- **核心对象是什么**：项目、监控任务、自动追踪规则、发布内容。
- **处于链路哪个位置**：主链路中的发布后效果监控阶段。
- **和前后模块怎么衔接**：前接合作沟通与执行，后接复盘、报告和支付依据。
- **最容易误判的点**：不要把它与品牌分析混淆；内容监控看的是“项目/内容级持续追踪”，品牌分析看的是“品牌级市场与策略洞察”。

### 15. 支付中心

- **解决什么业务问题**：把营销合作的资金流、支付单和草稿管理集中处理。
- **核心对象是什么**：账户余额、支付单、收支明细、草稿箱。
- **处于链路哪个位置**：主链路收尾的支付结算阶段。
- **和前后模块怎么衔接**：前接合作执行与确认，后接财务追踪与账户管理。
- **最容易误判的点**：支付中心不是套餐购买页；`/product/pricing` 是公开转化页，`/payment/list` 是工作台内的业务支付模块。

### 16. 商品中心 / 短链 / 物流追踪

- **解决什么业务问题**：围绕商品协同、链接追踪和寄样物流形成执行支持。
- **核心对象是什么**：商品、短链、物流单号。
- **处于链路哪个位置**：主链路执行后段的商品协同辅助层。
- **和前后模块怎么衔接**：可与达人合作、带货场景、效果统计共同使用。
- **最容易误判的点**：它和 TikTok Shop 搜索不是一回事；前者偏内部运营协同，后者偏外部情报发现。

### 17. 数据用量与配额

- **解决什么业务问题**：明确每种高成本能力的使用量、剩余额度与历史记录。
- **核心对象是什么**：查看邮箱额度、品牌详情额度、导出额度、监控额度、自动追踪额度等。
- **处于链路哪个位置**：主链路收尾之后的治理与成本控制层。
- **和前后模块怎么衔接**：为邮件、品牌分析、监控、导出等多个业务模块提供横向约束。
- **最容易误判的点**：`service/*` 页面不是独立业务产品，而是多模块共享的服务台账层。

### 18. 通知、账号安全、团队权限、Skills

- **解决什么业务问题**：让多人协作、账号安全和新能力调用有统一治理入口。
- **核心对象是什么**：系统消息、密码与登录方式、团队成员、权限、API 密钥、账单、配额。
- **处于链路哪个位置**：平台治理与服务层。
- **和前后模块怎么衔接**：横向服务于所有业务模块，而不是某条单独链路的一个步骤。
- **最容易误判的点**：不要把 Skills 控制台当成传统帮助中心；它更像面向新能力与 API Key 的独立平台面板。

---

## 最终判断

Nox 更准确的理解方式不是“一个大搜索站 + 若干零散后台”，而是：

> **以前台公开入口获客，以搜索/品牌情报发现机会，以 CRM/邮件/消息/监控/支付完成执行，再用配额/权限/Skills 做平台治理。**

这也是后续做任何模块评价时最应该先带上的整体认知。
