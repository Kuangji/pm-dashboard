## 维度 1：功能定义

**解决的用户问题**：

「我是品牌方/代理商，想知道谁在社媒上提到/传播这个品牌词，这些内容的覆盖规模是多少，以及相关达人的结构分布是怎样的。」

用 JTBD 格式表达：

> **When** 我要做品牌舆情监控或达人数据库建立，
> **I want to** 输入品牌关键词，直接看到全平台的内容和达人清单，
> **So that** 我可以评估品牌在社媒上的声量、发现潜在合作达人、追踪竞品动态。

**覆盖的核心场景**：

| 场景 | 典型用户 | 使用路径 |
|------|---------|---------|
| 品牌声量监控 | 品牌市场团队 | 搜索品牌词 → Posts 数据 → Post Insight 趋势 |
| 达人筛选/入库 | 品牌合作运营 | 搜索品牌词 → Creator List → 按粉丝/KBI 筛选 |
| 竞品动态监测 | 竞品分析岗 | 搜索竞品词 → 对比自身品牌词数据 |
| 品类/话题研究 | 社媒内容策划 | Post Insight → Hashtag 词云 + 趋势折线 |

**功能边界（这个功能不解决的）**：

- 不提供内容创作建议（只是数据聚合，不做内容推荐）
- 不区分同名品牌（「Dreame 家电」vs「Dreame 小说平台」——系统无消歧义能力）
- 不做跨平台账号合并（同一达人在 TikTok 和 Instagram 分别计为两条记录）

## 维度 2：交互实现

**UI 结构**：

```
左侧导航（10 项）
├── Quick Search（首页）
│   ├── 搜索框（Post/Creator 双 Tab）
│   └── Trending 板块（Songs / Hashtags / Creators 三列，默认地区：美国，近 7 天）
│
├── Posts Analytics
│   ├── 搜索框（Filter 漏斗图标 + My Saved Searches 下拉）
│   ├── 平台分布 checkbox（10 个平台，带实时数量）
│   ├── 时间筛选（默认近 7 天，⚠️ 标注不显著）
│   ├── Post List Tab（表格视图）
│   └── Post Insight Tab（词云 + 趋势图 + 数据表）
│
└── Creators Analytics
    ├── 搜索框（Filter + My Saved Searches）
    ├── 平台分布 checkbox（10 个平台，带实时数量）
    ├── Creator List Tab（表格视图）
    └── Creator Insight Tab（品类分布图 + 气泡图 + 数据表）
```

## 维度 3：边界处理

**已观察到的限制场景**：

| 限制类型 | 具体表现 | 影响 |
|---------|---------|------|
| **子串匹配噪音** | 搜索「dreame」触发「dreamer/dreamers/dreamin」账号（91.4% Creators 结果为此类噪音）| 用户需要人工过滤，列表参考价值低 |
| **同名品牌混淆** | 「Dreame」同时对应家电品牌、小说平台、家具品牌（仅 Posts 有效率约 60%）| 搜索结果置信度存疑，需用户自行判断 |
| **导出上限 10,000 条** | UI 显示 165.6K 创作者，系统最多导出 10,000 条（占总量 6%）| 预期落差极大，影响用户信任度 |
| **Posts 默认时间范围近 7 天** | 搜索 Posts 时默认应用「过去 7 天」时间筛选，无显著视觉标注 | 用户误认为「3K」是全量，实际全量远大于 3K |
| **Creator 无时间筛选** | Creator 搜索展示全量历史账号（不限时间），数量庞大但无时间维度 | 无法区分「最近有活跃」vs「历史上提过一次」的达人 |

**未观察到（待 T8-T10 补充）**：

- 无结果时的空状态 UI
- 关键词过长时的截断/提示逻辑
- 搜索请求失败时的错误处理

**降级处理（推测）**：

无明显降级设计可见，系统在数据量极大时仍正常展示（165.6K 结果不分页加载，而是通过 checkbox 过滤减少渲染量）。

## 维度 4：业务信号

**反映的战略意图**：

- **定位「品牌社媒舆情监控」**：不是通用 KOL 发现工具，而是以「品牌词」为核心组织数据，功能围绕「这个词在各平台的声量和达人分布」展开
- **搜索→监测闭环**：搜索结果页直接提供「Add Report」入口，把一次性搜索转化为持续监测，提升用户粘性和 LTV
- **Insight Tab 的分析层升级**：从「给你一堆数据」到「帮你看懂趋势」，拉升功能价值感，支持更高定价
- **Quick Search + Trending 首页**：快速留住新用户（冷启动友好），不让首页空闲，用热榜驱动探索行为

**数据优势/壁垒**：

| 能力 | 壁垒程度 | 说明 |
|------|---------|------|
| KBI 影响力指数 | 高（自研指标）| 自建的综合影响力评分，客户依赖后有迁移成本 |
| 情感分析（Sentiment）| 中（需 NLP 积累）| Neutral/Positive/Negative 三分类，品牌监控高频需求 |
| 多平台覆盖（10 个）| 中（数据采集积累）| Website 内容源是差异化，其他竞品少见 |
| 历史全量 Creator 数据库 | 低（量大但质低）| 165K 中 91.4% 是噪音，「大数据库」优势存疑 |

## 维度 6：值得学习 vs 值得避开

**可复用模式**：

| 模式 | 具体描述 | 为什么值得学 |
|------|---------|------------|
| **List/Insight 双 Tab** | 搜索结果页 List Tab 和 Insight Tab 同级切换，不丢失搜索状态 | 同一搜索词下「明细」和「趋势」无缝切换，避免用户二次操作；适合 Nox 内容搜索/品牌监控场景 |
| **顶部平台 checkbox + 实时数量** | 搜索后顶部展示各平台内容数（如 TikTok 1.18K），点击即过滤 | 让用户在一处看到全平台分布，认知成本低；可直接移植到 Nox 搜索结果页 |
| **首页展示 Trending 热榜** | Quick Search 首页展示过去 7 天 Top Songs / Hashtags / Creators | 搜索页不闲置，用热榜内容驱动用户探索，降低冷启动摩擦；适用于 Nox 关键词搜索功能的空状态/首屏设计 |
| **Post Insight 三层可视化** | 词云（感性）→ 趋势折线（时间维度）→ 数据表（精确量化）三层递进 | 满足不同深度需求，从宏观感知到精确数字；适合 Nox 的内容聚合分析层设计 |
| **情感标签（Sentiment）** | 每条 Post 标注 Neutral/Positive/Negative | 品牌监控场景高频需求，帮助用户快速识别负面舆情；Nox 可作为 P1 功能跟进 |

**反模式**：

| 反模式 | 具体表现 | 根本问题 | Nox 的替代方案 |
|--------|---------|---------|--------------|
| **Creator 子串匹配** | 搜索「dreame」匹配「dreamer/dreamers」，10K 样本中 91.4% 是无关账号 | 子串匹配无界限，数量庞大但质量极低，用户被大数字误导 | 默认精确匹配，模糊扩展作为可选 opt-in；或搜索前提示「当前使用精确匹配模式」 |
| **大数字掩盖低质量** | UI 显示 165K Creator，实际有效约 2,700（1.6%），差距 60 倍 | 虚假优越感，客户在 Demo 对比时被误导（165K vs Nox 7K），可能导致错误购买决策 | Nox 在结果数量旁标注口径（「7,366 有效达人（有内容发布记录）」），主动透明化 |
| **同名品牌无消歧义** | 「Dreame 家电」「Dreame 小说平台」「Dreame 沙发」三个品牌混入同一结果 | 纯关键词命中，无行业/品类区分，品牌监控场景置信度存疑 | 行业/品类维度筛选（家电/家居/数字内容）+ 歧义词检测提示 |
| **默认时间范围无感知** | Posts 搜索默认「近 7 天」，时间标注不显著，用户误认为 3K 是全量 | 信息不对称导致用户误读数据，影响分析决策质量 | 结果数量旁明确标注时间口径（「3K（近 7 天）」），支持用户修改默认范围 |
| **导出量与展示量严重不符** | UI 显示 165K，导出上限 10K（仅 6%），预期落差极大 | 用户花时间筛选了 165K 的结果，导出时才发现只能拿到 6%，体验断崖 | 导出按钮旁前置提示「当前结果 X 条，最多可导出 Y 条」；或提供分批导出引导 |
