# Modash 功能地图

## 总览

从公开页面看，Modash 的产品结构可以拆成两层：

- **核心工作流层**：Discover → Manage / Inbox → Track → Pay
- **连接增强层**：Shopify integration / Email integration / API

这说明 Modash 的产品设计不是“功能点并列”，而是围绕一条 creator operation 流程来组织。

---

## 一级模块

| 模块 | 作用 | 在整条链路中的位置 |
|------|------|------|
| Discover | 搜索、筛选、建立 shortlist | 前门 |
| Manage | 管理 creator 关系、状态、列表与流程 | 中控 |
| Inbox / Email | 承接实际沟通 | 执行 |
| Track | 跟踪内容上线与 campaign 表现 | 反馈闭环 |
| Pay | 处理发票、付款、税务与合规 | 收尾闭环 |
| Shopify / API | 连接电商与内部系统 | 放大器 |

---

## Discover 层

公开卖点包括：

- 大规模创作者库
- 受众画像筛选
- 假粉 / audience quality
- past brand partnerships
- AI visual search
- lookalike
- public email

**角色**：这是获客入口，也是最容易被市场理解的模块。  
**判断**：Discover 是 Modash 的“第一眼卖点”，但不是唯一卖点。

---

## Manage / Inbox 层

从 G2 用户反馈与官方导航推断，Manage / Inbox 解决的问题是：

- 把 creator 关系与状态放进统一系统
- 让团队能看见合作上下文
- 减少对个人记忆、个人邮箱和离散文档的依赖

**角色**：这是把 Modash 从“搜索工具”抬升为“工作流平台”的关键。  
**判断**：如果没有这一层，Modash 的差异化会弱很多。

---

## Track 层

用户特别提到的点包括：

- 自动发现品牌相关内容
- 跟踪 campaign 内容是否上线
- 看到 performance 与 campaign flow

**角色**：Track 是承上启下的模块，既让 discovery 结果闭环，也让后续付款有依据。  
**判断**：Track 不是最性感的模块，却是 workflow 完整性的关键。

---

## Pay 层

公开卖点里，Modash 把 Pay 与税务、合规、发票来回沟通并列呈现，明显是为了扩大产品价值边界。

**角色**：把“找人合作”延伸到“把合作跑完”。  
**判断**：Pay 是 Modash 区别于纯 discovery / analytics 平台的重要标志，但在社区里仍被视作有边界的模块。

---

## 连接增强层

### Shopify integration

这说明 Modash 不是泛化营销平台，而是明显把自己嵌入了电商品牌的 creator 运营链路。

### Email integration

说明官方知道 outreach 不能只停在平台内，而要尽量把真实沟通也纳入同一工作流。

### API

说明它也在争取更成熟团队或生态伙伴，不满足于完全封闭 SaaS。

---

## 最终判断

Modash 的功能地图不是“把所有营销能力都做一遍”，而是围绕一条高频路径做深：

> **找到人 → 判断值不值得合作 → 发起沟通 → 跟踪内容 → 结算付款**

这条路径正是它相对 Nox 和头部平台最值得对照的主线。
