# YouTube to NotebookLM · 调研简报

## Phase 0：决策锚定

| 维度 | 结论 |
|---|---|
| 决策目标 | 判断 Chrome 插件 `YouTube to NotebookLM` 的真实能力、技术实现与限制，并评估 NoxInfluencer 是否应复制，以及应复制到什么形态 |
| 功能边界 | 聚焦 `单视频 / 播放列表 / 频道 / 搜索结果 -> 批量写入 AI 研究容器`；尤其关注 `整频道数据获取`。不展开评估 NotebookLM 全产品能力 |
| 竞品候选 | 直接拆解对象：`YouTube to NotebookLM`；邻域参照：`NotebookLM 官方 YouTube URL 支持`、`NotebookLM 移动端原生分享入口`、`相关 NotebookLM 工具扩展` |
| 采集模式 | 全部为 `[自动]`：Chrome Web Store、Google Docs 隐私政策、Google 官方博客/帮助、Reddit 社区、CRX 包源码 |
| 产出用途 | ✅ 可直接作为 Nox 后续 `input / spec / design` 的上游输入，尤其适用于 `频道 AI 研究工作台`、`站内知识库`、`浏览器轻入口` 三类方向 |

## 基于本次任务默认采用的前提

- 用户更看重 `整频道数据获取能力`，而不是插件 UI 本身。
- 研究重点不是“要不要对接 NotebookLM”，而是“如何让 Nox 把整频道内容送入 AI 可问答容器”。
- 结论必须同时回答：
  - 这个插件到底怎么拿频道数据
  - 它是不是在吹“整个频道”
  - 哪些能力能在 Nox 场景中更稳、更深地复制

## 关键待回答问题

1. 这个插件拿 YouTube 视频列表时，依赖的是官方 API、页面 DOM，还是 YouTube 前台私有接口？
2. 它把内容写进 NotebookLM 时，走的是公开能力还是私有 RPC？
3. “整个频道”在真实实现里到底意味着 `全量`、`最近 N 条`，还是 `当前页面可见部分`？
4. 在 NoxInfluencer 语境下，最值得复制的是：
   - 浏览器轻入口
   - 整频道抓取链路
   - 站内 AI 知识工作台
   - 还是三者的组合
